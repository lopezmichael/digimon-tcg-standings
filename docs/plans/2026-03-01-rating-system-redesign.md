# Rating System Redesign

**Date:** 2026-03-01
**Status:** Approved
**Author:** Claude (with Michael)

## Overview

Redesign the competitive rating (Elo) calculation to use a single-pass chronological approach instead of the current 5-pass iteration. This fixes several bugs and enables proper handling of backfilled tournaments.

## Problems with Current System

### 1. Multi-Pass Iteration Causes Global Recalculation
The current algorithm iterates through all tournaments 5 times for "convergence." This means adding any tournament causes ALL players' ratings to shift, not just players in that tournament.

### 2. Anachronistic Dependencies
On pass 2+, a January tournament uses ratings that already include February/March data from pass 1. Future information affects past calculations.

### 3. Decay Anchored to Current Date
```r
months_ago <- difftime(Sys.Date(), tourney$event_date) / 30.44
decay_weight <- 0.5 ^ (months_ago / 4)
```
This applies decay based on tournament age, not rating staleness. A January tournament always gets less weight than a March tournament, regardless of when you calculate.

### 4. Ties Treated as Losses
```r
actual_result <- if (placement < opponent_placement) 1 else 0
```
Equal placements (ties) return 0, counting as a loss. Should be 0.5.

### 5. No Incremental Updates
Cannot add a tournament and only recalculate affected players/dates. Must rebuild everything.

## New Design

### Single-Pass Chronological Algorithm

Process tournaments exactly once, in strict date order:

1. **Initialize:** All players start at 1500
2. **Process tournaments in date order:** Each tournament uses ratings as they exist at that moment
3. **Update immediately:** Rating changes are applied before processing the next tournament
4. **Record history:** Store rating before/after each tournament for each player

### Backfill Handling

When a tournament is added with date X:
1. Find all tournaments from date X forward
2. Delete rating history from date X forward
3. Get players' ratings as of (X - 1 day) from history, or 1500 if no prior history
4. Single-pass recalculate from date X forward
5. Update player_ratings_cache with final values

This handles the "cascade" effect — if backfilling affects Player A, and Player A was in later tournaments with Players B, C, D, those are all recalculated too.

### Tie Fix

```r
actual_result <- if (placement < opponent_placement) 1
                 else if (placement == opponent_placement) 0.5
                 else 0
```

### Decay Removed

Standard Elo doesn't use time-based decay. A player who was 1800 six months ago and hasn't played is still 1800 — their skill is assumed unchanged until proven otherwise.

**Future consideration:** De-emphasize inactive players in leaderboard display (filter/visual treatment), not in the rating calculation itself.

### Modifiers Retained

| Modifier | Formula | Purpose |
|----------|---------|---------|
| K-factor | 48 (< 5 events) / 24 (established) | New players' ratings move faster |
| Round multiplier | `min(1.0 + (rounds - 3) * 0.1, 1.4)` | More rounds = more signal |

## Schema Changes

### New Table: player_rating_history

```sql
CREATE TABLE player_rating_history (
  history_id INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  player_id INTEGER NOT NULL REFERENCES players(player_id),
  tournament_id INTEGER NOT NULL REFERENCES tournaments(tournament_id),
  rating_before INTEGER NOT NULL,
  rating_after INTEGER NOT NULL,
  rating_change INTEGER NOT NULL,
  events_played INTEGER NOT NULL,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,

  UNIQUE(player_id, tournament_id)
);

CREATE INDEX idx_rating_history_player ON player_rating_history(player_id);
CREATE INDEX idx_rating_history_tournament ON player_rating_history(tournament_id);
```

## Implementation Steps

1. **Build comparison tool** — Capture current ratings before any changes
2. **Create migration** — Add player_rating_history table
3. **Implement new algorithm** — `calculate_ratings_from_date()`
4. **Run full rebuild** — Populate history, update cache
5. **Compare results** — Review before/after for specific players
6. **Update triggers** — Modify tournament add/edit/delete to call new recalc

## Rollback Plan

If issues arise:
1. Drop player_rating_history table
2. Revert R/ratings.R to previous version
3. Run old `recalculate_ratings_cache()` to restore previous ratings

## Success Criteria

- [ ] Adding a recent tournament only changes ratings for players from that date forward
- [ ] Backfilling an old tournament properly cascades through later tournaments
- [ ] Tied placements result in 0.5, not 0
- [ ] Rating history is recorded and queryable per player
- [ ] Comparison shows reasonable changes (no wild swings for established players)
